# 第二部分：信任边界检测

## 专题一：信任边界检测

### 2.1 技术介绍

信任边界是系统中可信和不可信数据之间的分界线。信任边界绕过是指攻击者将不可信数据作为可信数据处理，从而导致安全漏洞。

**信任边界概念：**
- **数据信任边界：** 数据从不可信源进入系统的边界
- **代码信任边界：** 代码执行权限变化的边界
- **网络信任边界：** 网络区域之间的边界
- **用户信任边界：** 不同用户角色之间的边界

**常见信任边界问题：**
- 未验证的用户输入被当作可信数据
- 客户端验证被当作服务端验证
- 内部服务调用未进行认证授权
- 第三方数据未经验证直接使用

### 2.2 信任边界识别

#### 2.2.1 信任边界类型

| 边界类型 | 描述 | 常见位置 |
|---------|------|---------|
| **用户输入边界** | 用户数据进入系统 | HTTP 参数、表单、API |
| **网络边界** | 不同网络区域之间 | 防火墙、负载均衡器 |
| **服务边界** | 不同服务之间 | API 网关、微服务 |
| **数据源边界** | 外部数据进入 | 文件导入、第三方 API |
| **权限边界** | 权限变化点 | 角色切换、特权操作 |
| **代码边界** | 代码执行上下文 | eval、反序列化 |

#### 2.2.2 信任边界识别方法

**数据流分析：**

```
用户输入 → 输入验证 → 业务逻辑 → 数据存储
    ↑           ↑          ↑         ↑
  不可信     边界点      可信区     可信区
```

**识别步骤：**

1. **识别所有输入点**
   ```bash
   # HTTP 输入点
   - URL 参数 (?key=value)
   - POST 数据
   - HTTP 头
   - Cookie
   - 文件上传
   
   # 其他输入点
   - WebSocket 消息
   - 消息队列
   - 数据库查询结果
   - 缓存数据
   - 第三方 API 响应
   ```

2. **标记信任级别**
   ```
   输入源          默认信任级别
   --------------------------
   用户输入        不可信
   Cookie          不可信
   内部 API        部分可信
   数据库数据      部分可信
   硬编码配置      可信
   ```

3. **追踪数据流**
   ```
   用户输入 → 参数验证？→ 业务处理 → 数据库
              ↓
           未验证 → 漏洞
   ```

### 2.3 信任边界绕过检测

#### 2.3.1 用户输入绕过

**检测方法：**

```bash
# 1. 测试输入验证
# 尝试绕过客户端验证

# 2. 测试类型转换
# 字符串转数字、布尔值转换

# 3. 测试特殊字符
# SQL 注入、XSS、命令注入

# 4. 测试边界值
# 负数、超大数、空值
```

**常见绕过技术：**

| 技术 | 示例 | 说明 |
|-----|------|------|
| **参数篡改** | `is_admin=false` → `is_admin=true` | 布尔值绕过 |
| **类型混淆** | `"123abc"` 转为数字 | 类型转换问题 |
| **数组绕过** | `{"role": "user"}` → `{"role": ["user", "admin"]}` | 数组处理差异 |
| **Unicode 绕过** | 使用同形字符 | 字符编码绕过 |
| **编码绕过** | URL 编码、Base64 | 编码解码差异 |

#### 2.3.2 客户端验证绕过

**检测方法：**

```bash
# 1. 识别客户端验证
# - JavaScript 验证
# - 移动端本地验证

# 2. 绕过客户端验证
# 直接使用 API 调用
curl -X POST https://target.com/api/action \
  -H "Content-Type: application/json" \
  -d '{"amount": -1000}'

# 3. 比较客户端和服务端逻辑
# 检查是否有差异
```

**常见场景：**

```javascript
// 客户端验证（可绕过）
if (amount > 0 && amount <= 1000) {
  submit();
}

// 服务端必须有相同验证
if (amount <= 0 || amount > 1000) {
  return error;
}
```

#### 2.3.3 内部服务信任绕过

**检测方法：**

```bash
# 1. 识别内部服务
# - 服务发现
# - API 文档

# 2. 测试服务间认证
# 直接从外部调用内部 API
```

**常见场景：**



#### 2.4.2 自动化检测脚本

```python
#!/usr/bin/env python3
"""
信任边界检测脚本示例
"""

import requests

def test_trust_boundary(url, test_cases):
    """测试信任边界"""
    for case in test_cases:
        response = requests.post(url, json=case)
        
        # 检查响应
        if response.status_code == 200:
            print(f"[+] 可能的信任边界绕过：{case}")
        else:
            print(f"[-] 验证正常：{case}")
```

### 2.5 信任边界加固

#### 2.5.1 输入验证最佳实践

```python
# 服务端验证所有输入
def validate_input(data):
    # 类型验证
    if not isinstance(data['amount'], (int, float)):
        raise ValueError("Invalid type")
    
    # 范围验证
    if data['amount'] <= 0 or data['amount'] > 10000:
        raise ValueError("Out of range")
    
    # 格式验证
    if not re.match(r'^[a-zA-Z0-9]+$', data['username']):
        raise ValueError("Invalid format")
    
    return True
```

#### 2.5.2 信任边界设计原则

```
1. 零信任原则
   - 默认不信任任何输入
   - 所有输入必须验证

2. 纵深防御
   - 多层验证
   - 客户端 + 服务端验证

3. 最小权限
   - 代码使用最小权限
   - 服务间最小必要信任

4. 明确边界
   - 清晰定义信任边界
   - 边界处严格验证
```

#### 2.5.3 代码示例

```python
# 危险模式：信任客户端数据
@app.route('/api/update_profile', methods=['POST'])
def update_profile():
    user_id = request.json['user_id']  # 危险：直接使用
    new_role = request.json['role']    # 危险：直接使用
    db.update_user(user_id, role=new_role)

# 安全模式：不信任客户端
@app.route('/api/update_profile', methods=['POST'])
def update_profile():
    # 从会话获取当前用户
    current_user = get_current_user()
    
    # 验证权限
    if not current_user.can_update_profile():
        return error
    
    # 忽略客户端传入的 user_id，使用会话中的
    # 不允许客户端设置角色
    db.update_user(current_user.id, data=request.json)
```

## 3.2 常见信任边界漏洞

| 漏洞 | 描述 | 风险等级 |
|-----|------|---------|
| **越权访问** | 未验证资源访问权限 | 高 |
| **参数篡改** | 修改关键参数 | 高 |
| **客户端绕过** | 绕过客户端验证 | 高 |
| **内部服务暴露** | 内部服务无认证 | 严重 |
| **数据注入** | 第三方数据未验证 | 高 |
| **会话固定** | 会话 ID 未重新生成 | 中 - 高 |
| **CSRF** | 未验证请求来源 | 中 - 高 |

## 3.3 信任边界矩阵

| 数据源 | 默认信任级别 | 验证要求 |
|-------|------------|---------|
| 用户输入 | 不可信 | 严格验证 |
| Cookie | 不可信 | 签名验证 |
| 内部 API | 部分可信 | 认证验证 |
| 数据库 | 部分可信 | 完整性验证 |
| 缓存 | 部分可信 | 有效期验证 |
| 第三方 API | 不可信 | 签名验证 |
| 配置文件 | 可信 | 完整性验证 |
| 硬编码 | 可信 | 代码审查 |

## 3.4 防御建议

1. **零信任**：默认不信任任何外部输入
2. **服务端验证**：所有输入必须在服务端验证
3. **明确边界**：清晰定义和记录信任边界
4. **数据验证**：验证所有跨越边界的数据
5. **认证授权**：服务间实施认证授权
6. **最小权限**：使用最小必要权限
7. **日志审计**：记录边界跨越事件
8. **定期审查**：定期审查信任边界设计
