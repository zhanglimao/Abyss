# 第一部分：文档概述

## 1.1 编写目的
- 为渗透测试人员和安全团队提供第三方供应商安全评估的系统化方法
- 指导组织评估开源项目和商业供应商的安全成熟度
- 帮助理解软件供应链上游风险的评估和缓解策略

## 1.2 适用范围
- 适用于选择第三方组件和库的决策过程
- 适用于供应商安全风险评估场景
- 适用于开源项目采用前的安全审查
- 适用于供应链安全合规要求（如行政命令 14028）

## 1.3 读者对象
- 安全工程师
- 采购官员
- 技术负责人
- 架构师

# 第二部分：核心评估技术专题

## 专题一：供应商安全评估

### 2.1 技术介绍

供应商安全评估是指对软件供应链上游供应商（开源项目维护者、商业软件供应商）的安全实践、可信度和风险进行系统性评估的过程。评估结果用于指导组件选择、风险缓解和供应商管理决策。

**供应商风险来源：**

```
┌─────────────────────────────────────────────────────────────┐
│                    供应商风险来源                            │
├─────────────────────────────────────────────────────────────┤
│  技术风险                                                   │
│  ├── 代码质量差，存在安全漏洞                               │
│  ├── 缺乏安全测试和代码审查                                 │
│  ├── 依赖管理不当，使用脆弱依赖                             │
│  └── 更新机制不安全                                         │
├─────────────────────────────────────────────────────────────┤
│  维护风险                                                   │
│  ├── 项目停止维护（abandoned）                              │
│  ├── 维护者数量不足                                         │
│  ├── 响应漏洞速度慢                                         │
│  └── 文档和沟通不足                                         │
├─────────────────────────────────────────────────────────────┤
│  供应链风险                                                 │
│  ├── 供应商自身供应链被攻陷                                 │
│  ├── 维护者账户被盗                                         │
│  ├── 内部人员威胁                                           │
│  └── 地缘政治风险                                           │
└─────────────────────────────────────────────────────────────┘
```

**评估维度：**

| 维度 | 评估内容 | 权重 |
|-----|---------|------|
| 项目健康度 | 活跃度、社区规模、更新频率 | 25% |
| 安全实践 | 安全测试、代码审查、漏洞响应 | 30% |
| 代码质量 | 测试覆盖率、文档完整性 | 20% |
| 供应商信誉 | 历史安全记录、行业评价 | 15% |
| 合规认证 | ISO 27001、SOC 2 等 | 10% |

### 2.2 评估常见于哪些业务场景

| 业务场景 | 功能示例 | 评估重点 |
|---------|---------|---------|
| 新技术选型 | 选择框架/库 | 项目健康度、安全实践 |
| 供应商采购 | 采购商业软件 | 合规认证、安全记录 |
| 开源组件采用 | 引入开源依赖 | 维护状态、漏洞历史 |
| 关键系统选型 | 核心基础设施组件 | 全面深度评估 |
| 合规审计 | 满足监管要求 | 文档和证据收集 |
| 风险评估 | 现有供应商风险审查 | 持续监控指标 |

### 2.3 开源项目评估方法

#### 2.3.1 项目健康度评估

**检查清单：**

```bash
# 1. 活跃度检查
# - 最近提交时间（应<3 个月）
# - 提交频率（每周应有提交）
# - Issue 响应时间（应<1 周）

# GitHub API 检查
curl https://api.github.com/repos/org/project | jq '{
  updated_at: .updated_at,
  pushed_at: .pushed_at,
  open_issues: .open_issues_count,
  stargazers: .stargazers_count,
  forks: .forks_count
}'

# 2. 维护者检查
# - 维护者数量（应>1）
# - 维护者活跃度
# - 是否有组织背书

curl https://api.github.com/repos/org/project/contributors | jq 'length'

# 3. 版本发布检查
# - 发布频率
# - 是否有稳定版本
# - 版本命名是否规范

curl https://api.github.com/repos/org/project/releases | jq '.[0:5] | .[] | {tag_name, published_at}'
```

**风险信号：**

| 信号 | 风险等级 | 说明 |
|-----|---------|------|
| 最后提交>1 年前 | 高 | 项目可能已废弃 |
| 仅 1 个维护者 | 中 | 单点故障风险 |
| Issue 积压>100 | 中 | 维护能力不足 |
| 无稳定版本 | 中 | 代码可能不成熟 |
| 无安全政策 | 中 | 安全意识不足 |

#### 2.3.2 安全实践评估

**检查清单：**

```bash
# 1. 安全政策检查
# 检查是否有 SECURITY.md 或安全政策文档
curl https://raw.githubusercontent.com/org/project/main/SECURITY.md
curl https://raw.githubusercontent.com/org/project/main/SECURITY.rst

# 2. 漏洞响应检查
# - 是否有漏洞报告流程
# - 历史漏洞响应时间
# - 是否有安全公告

# 3. 安全测试检查
# - 是否有 CI 安全扫描
# - 是否有依赖检查
# - 是否有模糊测试

# 检查 GitHub Actions 工作流
curl https://raw.githubusercontent.com/org/project/main/.github/workflows/ci.yml | grep -E "security|audit|scan"

# 4. 代码审查检查
# - PR 是否需要审查
# - 审查覆盖率
# - 是否有自动化检查
```

**安全实践评分表：**

| 实践 | 有 | 部分 | 无 |
|-----|---|-----|---|
| 安全政策文档 | 10 分 | 5 分 | 0 分 |
| 漏洞报告流程 | 10 分 | 5 分 | 0 分 |
| CI 安全扫描 | 10 分 | 5 分 | 0 分 |
| 依赖漏洞检查 | 10 分 | 5 分 | 0 分 |
| 代码审查要求 | 10 分 | 5 分 | 0 分 |
| 分支保护 | 10 分 | 5 分 | 0 分 |
| 双因素认证 | 10 分 | 5 分 | 0 分 |
| 发布签名 | 10 分 | 5 分 | 0 分 |

#### 2.3.3 代码质量评估

```bash
# 1. 测试覆盖率检查
# 检查 CI 配置中的覆盖率要求
curl https://raw.githubusercontent.com/org/project/main/.github/workflows/ci.yml | grep -E "coverage|codecov"

# 2. 文档完整性
# - 是否有 README
# - 是否有 API 文档
# - 是否有使用示例

curl https://raw.githubusercontent.com/org/project/main/README.md | wc -l

# 3. 代码复杂度
# - 使用工具分析代码复杂度
# - 检查是否有代码质量报告

# 4. 依赖管理
# - 依赖数量是否合理
# - 依赖是否及时更新
# - 是否有锁定文件

# npm 项目检查
curl https://raw.githubusercontent.com/org/project/main/package.json | jq '.dependencies | length'
```

### 2.4 商业供应商评估方法

#### 2.4.1 合规认证检查

| 认证 | 说明 | 验证方法 |
|-----|------|---------|
| ISO 27001 | 信息安全管理体系 | 要求提供证书 |
| SOC 2 Type II | 安全控制审计 | 要求提供报告 |
| FedRAMP | 美国政府云安全 | 检查 FedRAMP 市场 |
| CSA STAR | 云安全联盟认证 | 检查 STAR 注册 |
| PCI DSS | 支付卡行业安全 | 要求提供 AOC |

#### 2.4.2 安全记录调查

```
调查内容：
1. 历史安全事件
   - 是否有公开的安全事件
   - 事件响应是否及时
   - 事后改进是否到位

2. 漏洞披露记录
   - CVE 数量
   - 平均修复时间
   - 严重漏洞处理

3. 行业评价
   - 客户评价
   - 分析师报告
   - 同行评价
```

**CVE 记录检查：**
```bash
# NVD 搜索
curl "https://services.nvd.nist.gov/rest/json/cves/2.0?vendorName=VendorName"

# 分析 CVE 趋势
# - 年度 CVE 数量
# - 严重性分布
# - 修复时间统计
```

#### 2.4.3 供应商问卷

**安全实践问卷示例：**

```
1. 安全组织
   □ 是否有专职安全团队
   □ 安全团队规模
   □ 安全负责人联系方式

2. 开发流程
   □ 是否有安全培训
   □ 是否有安全编码规范
   □ 是否有代码审查流程

3. 测试验证
   □ 是否有渗透测试
   □ 测试频率
   □ 是否有第三方审计

4. 事件响应
   □ 是否有应急响应流程
   □ 响应时间 SLA
   □ 是否有演练

5. 供应链管理
   □ 是否管理自身供应商
   □ 是否提供 SBOM
   □ 更新机制是否安全
```

### 2.5 持续监控方法

#### 2.5.1 监控指标

| 指标 | 监控频率 | 告警阈值 |
|-----|---------|---------|
| 新版本发布 | 每日 | 新版本可用 |
| 安全公告 | 每日 | 新公告发布 |
| CVE 披露 | 每日 | 新 CVE 关联 |
| 项目活跃度 | 每周 | 30 天无提交 |
| 维护者变更 | 每周 | 核心维护者离开 |
| 依赖漏洞 | 每日 | 新漏洞发现 |

#### 2.5.2 监控工具

```bash
# 1. GitHub 监控
# 使用 GitHub Notifications
# 或使用第三方工具

# 2. CVE 监控
# NVD API 轮询
curl "https://services.nvd.nist.gov/rest/json/cves/2.0?keywordSearch=product-name"

# 3. 依赖监控
# 使用 Dependabot、Snyk 等

# 4. 安全公告监控
# 订阅供应商安全邮件列表
# 监控供应商安全页面
```

#### 2.5.3 风险评分模型

```
风险评分 = 技术风险 × 0.4 + 维护风险 × 0.3 + 供应链风险 × 0.3

技术风险 = (漏洞数量 × 严重性权重) / 代码规模
维护风险 = (1 - 活跃度分数) × (1 - 维护者多样性分数)
供应链风险 = 供应商风险 + 依赖风险
```

# 第三部分：附录

## 3.1 供应商评估检查清单

### 开源项目评估

| 检查项 | 通过标准 | 状态 |
|-------|---------|------|
| 最后提交时间 | <3 个月 | ☐ |
| 维护者数量 | ≥2 | ☐ |
| 安全政策文档 | 存在 | ☐ |
| 漏洞报告流程 | 明确 | ☐ |
| CI 安全扫描 | 启用 | ☐ |
| 分支保护 | 启用 | ☐ |
| 测试覆盖率 | >70% | ☐ |
| 依赖更新 | 及时 | ☐ |

### 商业供应商评估

| 检查项 | 通过标准 | 状态 |
|-------|---------|------|
| ISO 27001 认证 | 有效证书 | ☐ |
| SOC 2 报告 | Type II | ☐ |
| 安全团队 | 专职 | ☐ |
| 渗透测试 | 年度 | ☐ |
| 应急响应 | 有流程 | ☐ |
| SBOM 提供 | 支持 | ☐ |
| 安全 SLA | 明确 | ☐ |
| 合同条款 | 包含安全责任 | ☐ |

## 3.2 风险评估矩阵

| 可能性\影响 | 低 | 中 | 高 |
|-----------|---|---|---|
| 高 | 中风险 | 高风险 | 严重风险 |
| 中 | 低风险 | 中风险 | 高风险 |
| 低 | 低风险 | 低风险 | 中风险 |

## 3.3 风险缓解策略

| 风险等级 | 缓解策略 |
|---------|---------|
| 严重风险 | 立即停止使用，寻找替代 |
| 高风险 | 制定迁移计划，实施额外控制 |
| 中风险 | 持续监控，定期审查 |
| 低风险 | 接受风险，常规监控 |
