# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 漏洞原理

更新劫持是指攻击者通过篡改软件更新过程，将恶意代码作为合法更新分发给用户。

**本质问题**：
- 更新包未签名或签名验证缺失
- 更新通道未加密
- 更新服务器可被劫持

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-494 | 下载代码 without 完整性检查 |
| CWE-345 | 数据真实性验证不足 |
| CWE-829 | 从不可信控制域包含功能 |

## 2.2 攻击常见于哪些业务场景

| 场景 | 风险点 | 潜在危害 |
|-----|-------|---------|
| 桌面应用 | 自动更新 | 恶意软件分发 |
| 移动应用 | 热更新机制 | 代码执行 |
| IoT 设备 | 固件更新 | 设备沦陷 |
| 浏览器扩展 | 扩展更新 | 用户监控 |
| 企业软件 | 集中更新 | 大规模感染 |

## 2.4 漏洞利用方法

### 2.4.1 更新服务器劫持

```bash
# 1. 劫持更新服务器 DNS
# 2. 搭建恶意更新服务器
# 3. 返回恶意更新包

# 恶意更新包内容：
# - 正常功能 + 后门
# - 完全恶意代码
# - 间谍软件
```

### 2.4.2 更新包篡改

```bash
# 1. 拦截更新请求
# 2. 下载原始更新包
# 3. 解包并注入恶意代码
# 4. 重新打包（如无签名验证）
# 5. 发送给客户端
```

### 2.4.3 版本回滚攻击

```bash
# 1. 发现旧版本存在漏洞
# 2. 修改版本检查响应
# 3. 返回旧版本为"最新版本"
# 4. 用户被诱导降级
# 5. 利用已知漏洞攻击
```

### 2.4.4 供应链污染

```
攻击流程：
1. 攻陷更新服务器
2. 在所有更新中植入后门
3. 所有用户自动下载执行
4. 大规模感染

影响：
- 所有用户被影响
- 难以追溯源头
- 信任链被破坏
```

### 2.4.5 固件更新攻击

```bash
# IoT 设备固件更新通常防护较弱

# 1. 拦截固件更新
# 2. 修改固件植入后门
# 3. 设备刷入恶意固件
# 4. 设备被永久控制
```

## 专题二：高级更新劫持技术

### 3.4 构建时更新劫持（SolarWinds 技术）

**攻击原理：**
```
┌─────────────────────────────────────────────────────────────┐
│              构建时更新劫持攻击流程                          │
├─────────────────────────────────────────────────────────────┤
│  1. 入侵软件供应商构建系统                                   │
│     - 获取构建服务器访问权限                                 │
│     - 潜伏观察构建流程                                       │
├─────────────────────────────────────────────────────────────┤
│  2. 修改构建过程                                             │
│     - 不修改源代码（绕过代码审查）                           │
│     - 在编译时注入恶意代码                                   │
│     - 签名在注入后应用，签名仍然有效                         │
├─────────────────────────────────────────────────────────────┤
│  3. 分发受污染的更新                                         │
│     - 通过官方更新渠道分发                                   │
│     - 用户收到"合法"签名更新                                 │
│     - 实际包含恶意后门                                       │
└─────────────────────────────────────────────────────────────┘
```

**检测方法：**
```bash
# 1. 比对不同构建的产物
# 如果相同源代码产生不同二进制，可能存在注入
diff build1/checksums build2/checksums

# 2. 可重现构建验证
# 使用相同源代码重新构建，比对产物
docker run --rm -v $(pwd):/src build-image
diff original_build rebuilt_build

# 3. 监控构建环境
# 记录构建过程中所有文件访问
auditctl -w /path/to/build -p wa -k build_monitor
```

### 3.5 增量更新劫持

**攻击原理：**
```
某些更新系统使用增量更新（只下载变更部分）

攻击者可以：
1. 分析增量更新算法
2. 构造恶意增量包
3. 应用于旧版本产生恶意新版本
```

**利用示例：**
```python
# 伪代码：构造恶意增量包
def create_malicious_patch(old_version, new_version, malicious_code):
    # 1. 分析正常增量
    normal_patch = diff(old_version, new_version)
    
    # 2. 修改增量包含恶意代码
    malicious_patch = normal_patch.copy()
    malicious_patch.add_section('backdoor', malicious_code)
    
    # 3. 重新计算校验和（如果算法有缺陷）
    malicious_patch.checksum = forge_checksum(malicious_patch)
    
    return malicious_patch
```

### 3.6 更新服务器 DNS 劫持

**攻击流程：**
```
1. 监控目标网络流量
2. 识别更新服务器域名
3. DNS 欺骗/劫持将域名指向恶意服务器
4. 提供恶意更新包
5. 如果无签名验证，攻击成功
```

**检测命令：**
```bash
# 检查 DNS 解析是否被劫持
nslookup update.target.com
dig update.target.com @8.8.8.8  # 使用公共 DNS 对比

# 检查更新服务器证书
openssl s_client -connect update.target.com:443 -showcerts

# 监控更新请求
tcpdump -i any -n port 80 or port 443 | grep update
```
