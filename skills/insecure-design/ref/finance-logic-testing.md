# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 金融业务逻辑漏洞原理

金融业务逻辑漏洞是指金融系统在资金流转、账户管理、交易处理等业务流程设计中的安全缺陷。

**核心业务环节**：
- 账户管理与认证
- 资金充值与提现
- 转账与支付
- 理财与投资
- 借贷与还款
- 交易与结算

**本质问题**：
- 金额计算精度缺陷
- 事务处理设计不足
- 并发控制设计缺陷
- 审计追踪设计缺失

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-840 | 业务逻辑缺陷 |
| CWE-200 | 信息泄露 |
| CWE-284 | 访问控制不当 |

## 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| 转账汇款 | 行内/跨行转账 | 金额篡改、重复转账 |
| 充值提现 | 银行卡充值/提现 | 金额篡改、重复提现 |
| 支付结算 | 在线支付 | 支付绕过、金额修改 |
| 理财购买 | 基金/理财产品 | 份额篡改、收益操纵 |
| 贷款申请 | 信用贷款 | 资料篡改、重复贷款 |
| 信用卡 | 账单/还款 | 账单篡改、还款绕过 |
| 外汇交易 | 货币兑换 | 汇率操纵、交易篡改 |
| 证券交易 | 股票买卖 | 价格操纵、交易绕过 |

## 2.3 漏洞发现方法

### 2.3.1 业务流程分析

**步骤 1：绘制资金流转图**

```
典型转账流程：
登录 → 选择转账 → 输入收款人 → 输入金额 → 
确认信息 → 身份验证 → 执行转账 → 结果通知
```

**步骤 2：识别关键数据流**

```
金额数据流：
输入金额 → 手续费计算 → 总额计算 → 余额检查 → 
扣款 → 入账 → 余额更新
```

**步骤 3：识别信任边界**

```
客户端 vs 服务端：
- 转账金额（应在服务端验证）
- 手续费计算（应在服务端）
- 余额检查（应在服务端）
- 交易密码（加密传输）
```

### 2.3.2 关键参数识别

```
转账请求中的关键参数：
- 收款人账号/ID
- 转账金额
- 手续费
- 币种
- 转账用途
- 交易密码
- 验证码
```

## 2.4 漏洞测试方法

### 2.4.1 金额篡改测试

```bash
# 场景：转账功能

# 1. 正常转账
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 100.00,
    "password": "******"
}

# 2. 篡改金额
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": -100.00,  # 负值
    "password": "******"
}

# 3. 超大金额
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 999999999.99,  # 超过余额
    "password": "******"
}

# 4. 精度测试
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 100.001,  # 三位小数
    "password": "******"
}
```

### 2.4.2 重复交易测试

```bash
# 场景：重复提交

# 1. 快速重复请求
POST /api/transfer
{"to": "A", "amount": 100}

POST /api/transfer
{"to": "A", "amount": 100}  # 相同请求

# 2. 检查是否重复扣款
# 余额是否扣减两次
# 收款人是否收到两笔

# 3. 检查交易流水号
# 是否生成不同流水号
```

### 2.4.3 并发交易测试

```bash
# 场景：并发转账

# 1. 余额 100 元，并发转账
POST /api/transfer {"to": "A", "amount": 100}
POST /api/transfer {"to": "B", "amount": 100}
POST /api/transfer {"to": "C", "amount": 100}

# 2. 检查最终余额
# 是否变为 -200（应失败两笔）

# 3. 并发提现
# 同时发起多笔提现请求
```

### 2.4.4 手续费绕过测试

```bash
# 场景：转账手续费

# 1. 手续费篡改
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 100,
    "fee": 0  # 修改为 0
}

# 2. 负手续费
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 100,
    "fee": -10  # 负值
}

# 3. 手续费精度
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 100,
    "fee": 0.001  # 异常精度
}
```

### 2.4.5 余额检查绕过测试

```bash
# 场景：余额验证

# 1. 透支测试
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 999999,  # 远超余额
    "password": "******"
}

# 2. 浮点数精度攻击
# 利用浮点数计算误差
POST /api/transfer
{
    "to_account": "6222021234567890",
    "amount": 0.0000001
}

# 3. 并发余额检查
# 在余额更新前发起新交易
```

### 2.4.6 交易状态测试

```bash
# 场景：交易状态机

# 1. 状态跳过
# 待确认 → 已完成（跳过处理中）
POST /api/transaction/update
{
    "tx_id": "123",
    "status": "completed"
}

# 2. 状态回退
# 已完成 → 待处理
POST /api/transaction/update
{
    "tx_id": "123",
    "status": "pending"
}

# 3. 状态并发修改
```

### 2.4.7 利息/收益测试

```bash
# 场景：理财收益

# 1. 收益计算篡改
POST /api/investment/claim
{
    "product_id": "123",
    "interest": 999999  # 篡改收益
}

# 2. 重复领取收益
POST /api/investment/claim
{"product_id": "123"}
# 成功后再次领取

# 3. 提前支取
# 未到期产品提前支取
# 检查是否扣除应扣费用
```

### 2.4.8 汇率操纵测试

```bash
# 场景：外汇兑换

# 1. 汇率篡改
POST /api/fx/exchange
{
    "from": "USD",
    "to": "CNY",
    "amount": 100,
    "rate": 10.0  # 篡改汇率（正常约 7.2）
}

# 2. 时间窗口利用
# 在汇率更新前锁定旧汇率
```

## 2.5 漏洞利用方法

### 2.5.1 浮点数精度攻击

```bash
# 利用浮点数计算误差

# 1. 小额累积
# 每次利用精度误差获取 0.0001 元
# 大量交易累积

# 2. 余额精度攻击
POST /api/transfer
{"to": "A", "amount": 0.0000001}
# 多次后余额异常
```

### 2.5.2 时间窗口攻击

```bash
# 利用系统处理延迟

# 1. 在余额更新前发起多笔交易
# 2. 在日切时点操作
# 3. 在系统维护时操作
```

### 2.5.3 跨系统套利

```bash
# 利用系统间数据不一致

# 1. A 系统充值 → B 系统提现
# 2. 在同步完成前操作
# 3. 利用汇率差异
```

# 第三部分：附录

## 3.1 金融业务逻辑测试检查清单

```
□ 金额是否在服务端验证
□ 是否有余额检查
□ 是否有并发控制
□ 是否有防重复机制
□ 手续费是否正确计算
□ 交易是否有唯一流水号
□ 交易是否可追溯
□ 是否有异常交易检测
□ 是否有日终对账
□ 是否有审计日志
```

## 3.2 常用工具

| 工具 | 用途 | 链接 |
|-----|------|------|
| Burp Suite | 请求篡改 | https://portswigger.net/burp |
| Turbo Intruder | 并发请求 | Burp 扩展 |
| SQLMap | SQL 注入测试 | https://sqlmap.org/ |

## 3.3 修复建议

1. **金额验证** - 所有金额在服务端验证
2. **事务处理** - 使用数据库事务确保一致性
3. **并发控制** - 使用锁或乐观锁
4. **防重机制** - 添加交易流水号防重
5. **精度处理** - 使用定点数处理金额
6. **审计追踪** - 完整记录交易流水
7. **异常检测** - 实时监控异常交易
8. **对账机制** - 定期对账确保一致性
