# 业务逻辑测试框架方法论

---

# 第一部分：文档概述

## 1.1 编写目的

本文档旨在为渗透测试人员提供一套标准化的业务逻辑测试框架，帮助系统性地发现和利用业务逻辑层面的安全缺陷。

## 1.2 适用范围

本文档适用于各类 Web 应用、移动应用、API 系统的业务逻辑安全测试。

## 1.3 读者对象

本文件主要面向执行渗透测试任务的安全工程师、安全分析师。

# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 业务逻辑测试框架原理

业务逻辑测试框架提供了一套系统化的方法，用于识别、测试和利用业务流程设计中的安全缺陷。

**核心测试维度**：
- 流程完整性测试
- 参数完整性测试
- 状态一致性测试
- 规则合规性测试

**本质目标**：
- 发现设计层面缺陷
- 验证业务规则执行
- 识别滥用可能性
- 评估业务风险

### 测试框架层次

```
测试层次：
┌─────────────────────────────────┐
│ L1: 业务流程测试                 │
│ - 端到端流程                     │
│ - 分支流程                       │
│ - 异常流程                       │
├─────────────────────────────────┤
│ L2: 功能模块测试                 │
│ - 输入验证                       │
│ - 输出编码                       │
│ - 业务规则                       │
├─────────────────────────────────┤
│ L3: 数据流测试                   │
│ - 数据完整性                     │
│ - 数据一致性                     │
│ - 数据权限                       │
├─────────────────────────────────┤
│ L4: 状态转换测试                 │
│ - 状态合法性                     │
│ - 转换条件                       │
│ - 并发状态                       │
└─────────────────────────────────┘
```

## 2.2 测试流程

### 2.2.1 准备阶段

**步骤 1：业务理解**

```
理解内容：
- 业务模式
- 盈利模式
- 用户角色
- 核心流程
- 关键资产
```

**步骤 2：流程映射**

```
绘制内容：
- 主业务流程图
- 子业务流程图
- 状态转换图
- 数据流图
```

**步骤 3：风险识别**

```
识别内容：
- 高价值操作
- 资金相关功能
- 敏感数据操作
- 权限变更功能
```

### 2.2.2 测试执行

**步骤 1：正常流程测试**

```
目的：建立基准
- 执行正常业务流程
- 记录请求/响应
- 理解参数含义
- 识别验证点
```

**步骤 2：异常流程测试**

```
目的：发现边界缺陷
- 跳过流程步骤
- 回退流程状态
- 并发执行流程
- 中断后恢复流程
```

**步骤 3：参数篡改测试**

```
目的：发现验证缺陷
- 修改数值参数
- 修改类型参数
- 修改枚举参数
- 添加/删除参数
```

**步骤 4：规则绕过测试**

```
目的：发现规则执行缺陷
- 绕过业务规则
- 绕过限制条件
- 绕过审批流程
- 绕过验证步骤
```

### 2.2.3 报告阶段

**步骤 1：风险评估**

```
评估维度：
- 可能性
- 影响程度
- 业务影响
- 合规影响
```

**步骤 2：修复建议**

```
建议内容：
- 技术修复方案
- 流程优化建议
- 监控检测建议
- 应急响应建议
```

## 2.3 测试方法库

### 2.3.1 流程测试方法

```
方法 1：步骤跳过测试
- 尝试跳过流程中的步骤
- 尝试直接访问后续步骤
- 尝试并行执行步骤

方法 2：步骤重排测试
- 改变步骤执行顺序
- 尝试逆序执行
- 尝试并发执行

方法 3：状态回退测试
- 尝试回退已完成步骤
- 尝试重用已使用资源
- 尝试恢复已取消状态

方法 4：流程分支测试
- 测试各分支条件
- 测试分支切换
- 测试分支并发
```

### 2.3.2 参数测试方法

```
方法 1：数值篡改
- 修改金额为 0 或负数
- 修改数量为超大值
- 修改价格为异常值
- 测试边界值

方法 2：类型篡改
- 字符串改为数字
- 数字改为字符串
- 布尔值取反
- 数组改为对象

方法 3：枚举篡改
- 使用未定义的枚举值
- 使用已废弃的枚举值
- 使用管理员枚举值
- 组合多个枚举值

方法 4：引用篡改
- 修改资源 ID
- 修改用户 ID
- 修改订单 ID
- 修改关联引用
```

### 2.3.3 规则测试方法

```
方法 1：条件绕过
- 不满足条件直接执行
- 伪造条件满足
- 修改条件判断结果
- 绕过条件检查

方法 2：限制绕过
- 绕过数量限制
- 绕过频率限制
- 绕过配额限制
- 绕过时间限制

方法 3：权限绕过
- 越权访问功能
- 越权访问数据
- 越权执行操作
- 越权修改配置

方法 4：审批绕过
- 跳过审批步骤
- 自审自批
- 伪造审批结果
- 修改审批流程
```

### 2.3.4 并发测试方法

```
方法 1：竞态条件测试
- 并发修改同一资源
- 并发使用同一凭证
- 并发执行互斥操作

方法 2：双重支付测试
- 并发发起支付
- 重放支付请求
- 修改支付状态

方法 3：超卖测试
- 并发购买限量商品
- 超额使用余额
- 超额使用优惠券

方法 4：TOCTOU 测试
- 检查和使用间修改
- 读取和写入间修改
- 验证和执行间修改
```

# 第三部分：附录

## 3.1 业务逻辑测试检查清单

```
□ 业务流程是否理解
□ 流程图是否绘制
□ 风险点是否识别
□ 正常流程是否测试
□ 异常流程是否测试
□ 参数篡改是否测试
□ 规则绕过是否测试
□ 并发测试是否执行
□ 风险评估是否完成
□ 修复建议是否提供
```

## 3.3 最佳实践

1. **深入理解业务** - 充分理解业务逻辑
2. **系统化测试** - 按照框架系统测试
3. **创造性思维** - 从攻击者角度思考
4. **详细记录** - 记录所有测试过程
5. **风险评估** - 准确评估业务风险
6. **可行建议** - 提供可行修复建议
