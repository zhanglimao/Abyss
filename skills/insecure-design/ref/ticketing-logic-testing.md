# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 票务业务逻辑漏洞原理

票务业务逻辑漏洞是指票务系统在座位选择、票务预订、支付出票等业务流程设计中的安全缺陷。

**核心业务环节**：
- 票务查询与展示
- 座位选择与锁定
- 订单创建与支付
- 出票与核销
- 退改签处理

**本质问题**：
- 库存并发控制缺陷
- 座位锁定机制不足
- 价格策略设计漏洞
- 退改签规则缺陷

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-840 | 业务逻辑缺陷 |
| CWE-770 | 资源分配无限制 |
| CWE-284 | 访问控制不当 |

## 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| 选座购票 | 座位选择 | 座位锁定绕过、重复选座 |
| 限时抢购 | 热门演出票 | 机器人抢票、超卖 |
| 价格策略 | 不同区域票价 | 区域价格篡改 |
| 优惠券 | 票务优惠 | 优惠券滥用 |
| 退改签 | 退票/改期 | 恶意退票、改签绕过 |
| 团体票 | 批量购买 | 批量囤票、倒卖 |
| 会员优先 | 会员专享购票 | 会员身份伪造 |
| 实名制 | 实名购票 | 身份信息绕过 |

## 2.3 漏洞发现方法

### 2.3.1 业务流程分析

**步骤 1：绘制购票流程图**

```
典型购票流程：
选择场次 → 选择座位 → 锁定座位 → 填写信息 → 
支付 → 出票 → 核销入场
```

**步骤 2：识别关键数据流**

```
座位状态流：
可选 → 已锁定（临时） → 已支付（已售） → 已核销

价格数据流：
基础票价 → 区域系数 → 优惠折扣 → 最终价格
```

**步骤 3：识别信任边界**

```
客户端 vs 服务端：
- 座位选择（应在服务端验证）
- 价格计算（应在服务端）
- 库存扣减（应在服务端）
- 身份信息（应验证真实性）
```

### 2.3.2 关键参数识别

```
购票请求中的关键参数：
- 场次 ID
- 座位 ID/排号/座号
- 票价
- 数量
- 优惠券 ID
- 身份信息
- 联系方式
```

## 2.4 漏洞测试方法

### 2.4.1 座位锁定绕过测试

```bash
# 场景：选座购票

# 1. 正常选座
POST /api/seat/lock
{
    "session_id": "SHOW001",
    "seat_ids": ["A1", "A2"]
}
# 锁定 15 分钟

# 2. 重复锁定同一座位
POST /api/seat/lock
{
    "session_id": "SHOW001",
    "seat_ids": ["A1"]  # 已被锁定
}
# 检查是否允许

# 3. 锁定后不支付
# 检查座位释放机制
```

### 2.4.2 超卖测试

```bash

        -d "session=SHOW001&seat=A$i" &
done

# 座位总数 100，检查是否超卖

# 2. 多账户同时购买
# 使用多个账户同时抢购

# 3. 锁定 + 支付并发
```

# 第三部分：附录

## 3.1 票务业务逻辑测试检查清单

```
□ 座位是否有锁定机制
□ 锁定是否有超时释放
□ 库存是否有并发控制
□ 价格是否在服务端计算
□ 限购规则是否有效
□ 退票是否有防重机制
□ 改签差价是否正确计算
□ 实名制是否有效验证
□ 核销是否有防重机制
□ 是否有异常购票检测
```

## 3.2 常用工具

| 工具 | 用途 | 链接 |
|-----|------|------|
| Burp Suite | 请求篡改 | https://portswigger.net/burp |
| Turbo Intruder | 并发请求 | Burp 扩展 |
| Selenium | 浏览器自动化 | https://selenium.dev/ |

## 3.3 修复建议

1. **座位锁** - 使用分布式锁处理选座
2. **库存控制** - 使用数据库锁或 Redis 原子操作
3. **价格验证** - 所有价格计算在服务端
4. **限购规则** - 多维度限购检查
5. **防重机制** - 关键操作添加防重
6. **实名验证** - 对接公安系统验证
7. **核销验证** - 一次性核销码
8. **风控系统** - 建立票务风控体系
