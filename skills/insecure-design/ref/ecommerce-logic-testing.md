# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 电商业务逻辑漏洞原理

电商业务逻辑漏洞是指电子商务平台在业务流程设计中的安全缺陷，攻击者可利用这些缺陷获取不当利益。

**核心业务环节**：
- 商品浏览与搜索
- 购物车管理
- 订单创建与支付
- 物流与配送
- 售后与退款

**本质问题**：
- 价格计算逻辑缺陷
- 库存管理设计不足
- 订单状态机设计缺陷
- 促销规则设计漏洞

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-840 | 业务逻辑缺陷 |
| CWE-200 | 信息泄露 |
| CWE-284 | 访问控制不当 |

## 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| 商品购买 | 价格计算、数量选择 | 价格篡改、数量绕过 |
| 购物车 | 商品添加/修改 | 价格注入、商品替换 |
| 优惠券 | 优惠券领取/使用 | 重复使用、叠加滥用 |
| 积分系统 | 积分获取/消费 | 积分篡改、刷分 |
| 秒杀活动 | 限时抢购 | 机器人抢购、超卖 |
| 拼团活动 | 多人拼单 | 虚假拼团、自拼 |
| 退款退货 | 退款申请 | 重复退款、虚假退货 |
| 会员体系 | 会员等级/权益 | 等级篡改、权益滥用 |

## 2.3 漏洞发现方法

### 2.3.1 业务流程分析

**步骤 1：绘制完整业务流程图**

```
典型电商流程：
浏览商品 → 加入购物车 → 确认订单 → 选择支付方式 → 
支付 → 商家发货 → 确认收货 → 评价
```

**步骤 2：识别关键数据流**

```
价格数据流：
商品详情页 → 购物车 → 订单确认页 → 支付页 → 支付网关

库存数据流：
商品详情页 → 下单扣减 → 支付确认 → 发货扣减
```

**步骤 3：识别信任边界**

```
客户端计算 vs 服务端验证：
- 商品总价（应在服务端）
- 运费计算（应在服务端）
- 优惠金额（应在服务端）
- 积分抵扣（应在服务端）
```

### 2.3.2 关键参数识别

```
结算请求中的关键参数：
- 商品 ID 和数量
- 单价和总价
- 优惠券 ID
- 积分使用数量
- 运费
- 收货地址 ID
- 支付方式
```

## 2.4 漏洞测试方法

### 2.4.1 价格篡改测试

```bash
# 场景：商品结算

# 1. 正常购买流程
POST /api/cart/add
{"product_id": 123, "quantity": 1}

POST /api/checkout
{"cart_id": "xxx"}
# 响应：{"total": 999.99}

# 2. 篡改价格
POST /api/checkout
{
    "cart_id": "xxx",
    "total": 0.01  # 修改为 0.01
}

# 3. 或者修改单价
POST /api/cart/update
{
    "item_id": "yyy",
    "price": 0.01  # 修改单价
}

# 4. 检查响应
# - 是否接受低价
# - 是否重新计算
# - 是否有告警
```

### 2.4.2 数量绕过测试

```bash
# 场景：限购商品

# 1. 测试限购规则
POST /api/cart/add
{"product_id": 123, "quantity": 1}  # 成功

POST /api/cart/add
{"product_id": 123, "quantity": 100}  # 检查是否限制

# 2. 拆分订单绕过
POST /api/cart/add
{"product_id": 123, "quantity": 5}
POST /api/cart/add
{"product_id": 123, "quantity": 5}
# 累计 10 件，检查是否限制

# 3. 多账户测试
# 使用不同账户购买限购商品
```

### 2.4.3 优惠券滥用测试

```bash
# 场景：优惠券使用

# 1. 重复使用测试
POST /api/checkout
{"coupon_code": "SAVE50"}
# 成功后再次使用同一优惠券

# 2. 叠加测试
POST /api/checkout
{
    "coupons": ["SAVE50", "NEWUSER", "VIP20"]
}

# 3. 条件绕过测试
# 满 100 减 20，订单金额 10 元
POST /api/checkout
{
    "items": [{"id": 1, "price": 10}],
    "coupon": "SAVE20"  # 满 100 可用
}

# 4. 优惠券遍历
# 遍历优惠券 ID 尝试未领取的优惠券
```

### 2.4.4 积分操纵测试

```bash
# 场景：积分系统

# 1. 积分篡改
POST /api/order/use-points
{
    "points": -100  # 负值
}

POST /api/order/use-points
{
    "points": 999999  # 超大值
}

# 2. 积分重复获取
# 完成同一任务多次

# 3. 积分转移滥用
POST /api/points/transfer
{
    "to_user": "attacker",
    "points": 10000  # 超过余额
}
```

### 2.4.5 库存超卖测试

```bash
# 场景：限量商品

# 1. 并发购买测试
for i in {1..100}; do
    curl -X POST https://target.com/api/buy \
        -d "product_id=limited&quantity=1" &
done

# 库存 10 件，检查是否超卖

# 2. 下单不支付
# 占用库存后不支付
# 检查库存释放机制

# 3. 取消订单重下
# 取消后库存是否恢复
```

### 2.4.6 订单状态测试

```bash
# 场景：订单状态机

# 1. 状态跳过测试
# 待支付 → 已发货（跳过支付）
POST /api/order/update
{
    "order_id": "123",
    "status": "shipped"
}

# 2. 状态回退测试
# 已完成 → 待支付
POST /api/order/update
{
    "order_id": "123",
    "status": "pending"
}

# 3. 状态并发测试
# 同时修改订单状态
```

### 2.4.7 退款欺诈测试

```bash
# 场景：退款功能

# 1. 重复退款
POST /api/refund
{"order_id": "123"}
# 成功后再次申请

# 2. 超额退款
POST /api/refund
{
    "order_id": "123",
    "amount": 9999  # 超过订单金额
}

# 3. 部分退款滥用
# 多次申请部分退款
# 累计超过订单金额
```

### 2.4.8 运费绕过测试

```bash
# 场景：运费计算

# 1. 运费篡改
POST /api/checkout
{
    "shipping_fee": 0  # 修改为 0
}

# 2. 包邮条件绕过
# 不满包邮条件但修改为包邮
POST /api/checkout
{
    "items": [{"price": 10}],  # 不满 99
    "shipping_fee": 0
}

# 3. 运费模板绕过
# 修改收货地址 ID 为包邮地区
```

## 2.5 漏洞利用方法

### 2.5.1 0 元购利用

```bash
# 组合多种漏洞实现 0 元购

# 1. 价格篡改为 0.01
# 2. 运费篡改为 0
# 3. 使用优惠券抵扣
# 4. 使用积分抵扣
# 最终实付 0 元或接近 0 元
```

### 2.5.2 薅羊毛利用

```bash
# 利用新用户优惠

# 1. 批量注册新账户
# 2. 每个账户领取新人优惠
# 3. 下单享受优惠
# 4. 多账户重复操作
```

### 2.5.3 超卖获利

```bash
# 利用超卖漏洞

# 1. 并发抢购限量商品
# 2. 成功购买后转卖
# 3. 平台无法发货需赔偿
```

# 第三部分：附录

## 3.1 电商业务逻辑测试检查清单

```
□ 价格是否在服务端重新计算
□ 数量是否有限购检查
□ 库存是否有并发控制
□ 优惠券是否验证有效性和适用范围
□ 积分是否正确扣减
□ 运费是否正确计算
□ 订单状态是否可跳过/回退
□ 退款是否有防重复机制
□ 是否有异常订单检测
□ 是否有刷单检测
```

## 3.3 修复建议

1. **服务端计算** - 所有金额计算在服务端进行
2. **库存锁** - 使用分布式锁处理并发
3. **状态机验证** - 严格验证状态转换
4. **防重机制** - 关键操作添加防重检查
5. **异常检测** - 监控异常订单行为
6. **审计日志** - 记录所有关键操作
7. **风控系统** - 建立业务风控体系
