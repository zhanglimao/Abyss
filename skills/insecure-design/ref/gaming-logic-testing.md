# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 游戏业务逻辑漏洞原理

游戏业务逻辑漏洞是指游戏系统在虚拟货币、道具交易、任务奖励、竞技排名等业务流程设计中的安全缺陷。

**核心业务环节**：
- 用户账户与角色
- 虚拟货币系统
- 道具获取与交易
- 任务与成就系统
- 竞技与排名系统
- 社交与公会系统

**本质问题**：
- 经济系统设计缺陷
- 数值计算设计漏洞
- 状态同步设计缺陷
- 反作弊设计不足

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-840 | 业务逻辑缺陷 |
| CWE-200 | 信息泄露 |
| CWE-284 | 访问控制不当 |

## 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| 充值系统 | 游戏币充值 | 金额篡改、重复充值 |
| 虚拟交易 | 道具买卖 | 价格篡改、道具复制 |
| 任务系统 | 日常任务/成就 | 任务完成伪造 |
| 抽卡系统 | 随机抽取 | 概率操纵、结果预测 |
| 竞技场 | PVP 对战 | 战斗结果伪造 |
| 排行榜 | 分数排名 | 分数篡改 |
| 公会系统 | 公会管理 | 权限绕过 |
| 邮件系统 | 游戏邮件 | 道具领取滥用 |

## 2.3 漏洞发现方法

### 2.3.1 业务流程分析

**步骤 1：绘制经济系统图**

```
典型游戏经济：
充值 → 游戏币 → 购买道具 → 使用/交易 → 
任务奖励 → 游戏币 → 提现/转移
```

**步骤 2：识别关键数据流**

```
货币数据流：
账户余额 → 消费扣减 → 奖励增加 → 转移变更

道具数据流：
获取 → 背包存储 → 使用/装备 → 交易转移 → 销毁
```

**步骤 3：识别信任边界**

```
客户端 vs 服务端：
- 货币计算（应在服务端）
- 战斗计算（应在服务端）
- 概率抽取（应在服务端）
- 道具验证（应在服务端）
```

### 2.3.2 关键参数识别

```
游戏请求中的关键参数：
- 用户 ID/角色 ID
- 货币类型和数量
- 道具 ID 和数量
- 任务 ID
- 战斗数据
- 交易信息
```

## 2.4 漏洞测试方法

### 2.4.1 充值篡改测试

```bash
# 场景：游戏币充值

# 1. 正常充值
POST /api/recharge
{
    "user_id": "123",
    "amount": 100,
    "currency_type": "CNY"
}
# 获得 100 游戏币

# 2. 金额篡改
POST /api/recharge
{
    "user_id": "123",
    "amount": 0.01,  # 修改为 0.01
    "currency_type": "CNY"
}

# 3. 负值测试
POST /api/recharge
{
    "user_id": "123",
    "amount": -100,  # 负值
    "currency_type": "CNY"
}

# 4. 重复充值
# 使用同一支付凭证多次充值
```

### 2.4.2 货币复制测试

```bash
# 场景：货币转移

# 1. 并发转移

# 余额 100，检查是否透支

# 2. 自我转移

# 检查是否增加货币

# 3. 取消交易
# 在交易完成前取消
# 检查货币是否回滚
```

# 第三部分：附录

## 3.1 游戏业务逻辑测试检查清单

```
□ 充值是否验证支付凭证
□ 货币计算是否在服务端
□ 是否有并发控制
□ 道具是否唯一标识
□ 任务完成是否验证
□ 抽卡概率是否服务端
□ 战斗结果是否服务端
□ 排行榜是否防作弊
□ 商城价格是否服务端
□ 是否有反作弊系统
```

## 3.3 修复建议

1. **服务端权威** - 关键计算在服务端
2. **事务处理** - 使用数据库事务
3. **并发控制** - 处理并发请求
4. **数据验证** - 严格验证所有输入
5. **日志审计** - 记录所有关键操作
6. **反作弊** - 建立反作弊系统
7. **异常检测** - 检测异常行为
8. **经济监控** - 监控游戏经济健康
