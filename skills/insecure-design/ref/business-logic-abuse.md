# 第一部分：文档概述

## 1.1 编写目的

本文档旨在为渗透测试人员提供一套标准化的业务逻辑滥用攻击检测与利用流程，帮助发现和利用设计层面的安全缺陷。

# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 漏洞原理

业务逻辑滥用是指攻击者利用应用程序业务流程设计中的缺陷，执行非预期的操作或获取不当利益。

**本质问题**：
- 业务流程设计未考虑恶意场景
- 缺少对异常行为的检测
- 过度信任客户端输入
- 状态机设计缺陷

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-840 | 业务逻辑缺陷 |
| CWE-200 | 信息泄露 |
| CWE-284 | 访问控制不当 |

## 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| 电商系统 | 购物车/结算 | 价格篡改、数量绕过 |
| 支付系统 | 转账/支付 | 金额篡改、重复支付 |
| 票务系统 | 订票/选座 | 批量预订、超卖攻击 |
| 优惠券系统 | 优惠券使用 | 重复使用、叠加滥用 |
| 积分系统 | 积分获取/使用 | 积分篡改、刷分 |
| 邀请系统 | 邀请奖励 | 自邀请、批量注册 |
| 退款系统 | 退款申请 | 重复退款、超额退款 |
| 拍卖系统 | 竞价/拍卖 | 恶意竞价、拍卖绕过 |

## 2.3 漏洞发现方法

### 2.3.1 业务流程分析

**步骤 1：绘制业务流程图**

```
正常购买流程：
浏览商品 → 加入购物车 → 结算 → 支付 → 确认订单 → 发货
```

**步骤 2：识别关键参数**

```
结算请求中的关键参数：
- 商品 ID
- 商品数量
- 单价
- 总价
- 优惠券 ID
- 运费
- 支付方式
```

**步骤 3：识别信任边界**

```
客户端可控 vs 服务端验证：
- 价格：应在服务端计算
- 数量：应验证库存和限购
- 优惠券：应验证有效性和适用范围
```

### 2.3.2 参数篡改检测

**价格篡改测试**

```bash
# 原始请求
POST /api/checkout
{
    "items": [{"id": 123, "quantity": 1}],
    "total": 99.99
}

# 篡改价格
POST /api/checkout
{
    "items": [{"id": 123, "quantity": 1}],
    "total": 0.01
}

# 检查响应：
# - 是否接受低价
# - 是否重新计算价格
# - 是否有其他验证
```

**数量篡改测试**

```bash
# 原始请求
POST /api/cart/add
{
    "product_id": 123,
    "quantity": 1
}

# 篡改数量
POST /api/cart/add
{
    "product_id": 123,
    "quantity": 9999
}

# 检查响应：
# - 是否有限购检查
# - 是否有库存检查
# - 是否有价格异常检测
```

### 2.3.3 状态转换测试

**步骤 1：识别状态机**

```
订单状态：
待支付 → 已支付 → 已发货 → 已完成
    ↓         ↓
  已取消   退款中
```

**步骤 2：测试状态跳过**

```bash
# 尝试直接从待支付跳到已完成
POST /api/order/complete
{
    "order_id": "12345",
    "status": "completed"  # 跳过支付
}
```

**步骤 3：测试状态回退**

```bash
# 尝试将已完成的订单回退到待支付
POST /api/order/update
{
    "order_id": "12345",
    "status": "pending"  # 尝试回退
}
```

## 2.4 漏洞利用方法

### 2.4.1 价格篡改利用

```bash
# 场景：电商结算

# 1. 正常流程获取请求格式
POST /api/checkout
{
    "items": [
        {"product_id": 1001, "quantity": 1, "price": 999.99}
    ],
    "coupon": "SAVE10",
    "total": 899.99
}

# 2. 篡改价格
POST /api/checkout
{
    "items": [
        {"product_id": 1001, "quantity": 1, "price": 0.01}
    ],
    "coupon": "SAVE10",
    "total": 0.01
}

# 3. 或者篡改总价
POST /api/checkout
{
    "items": [
        {"product_id": 1001, "quantity": 1}
    ],
    "total": 0.01  # 直接修改总价
}
```

### 2.4.2 优惠券滥用

```bash
# 场景：优惠券系统

# 1. 重复使用优惠券
POST /api/checkout
{
    "coupon_code": "SAVE50"
}
# 成功后再次使用同一优惠券

# 2. 优惠券叠加
POST /api/checkout
{
    "coupons": ["SAVE50", "NEWUSER", "VIP20"]  # 尝试叠加多个
}

# 3. 优惠券条件绕过
POST /api/checkout
{
    "items": [{"id": 123, "price": 10}],  # 不满 100
    "coupon": "SAVE20"  # 满 100 减 20
}
```

### 2.4.3 库存超卖攻击

```bash
# 场景：限量商品抢购

# 1. 并发请求
for i in {1..100}; do
    curl -X POST https://target.com/api/buy \
        -d "product_id=limited_item&quantity=1" &
done

# 2. 检查是否超卖
# 库存 10 件，但成功购买超过 10 次
```

### 2.4.4 竞争条件利用

```bash
# 场景：账户余额

# 1. 同时发起多笔转账
POST /api/transfer
{"to": "account1", "amount": 100}

POST /api/transfer
{"to": "account2", "amount": 100}

# 2. 余额只有 100，但两笔都成功
# 结果：余额变为 -100
```

### 2.4.5 邀请奖励滥用

```bash
# 场景：邀请好友得奖励

# 1. 自邀请
# 使用不同邮箱但相同设备/支付信息注册

# 2. 批量注册
# 自动化注册大量账户进行互邀

# 3. 奖励重复获取
# 同一被邀请人多次注册获取奖励
```

## 2.5 漏洞利用绕过方法

### 2.5.1 服务端验证绕过

**技巧 1：参数污染**

```bash
# 发送多个同名参数
POST /api/checkout
price=999.99&price=0.01

# 服务端可能使用第一个或最后一个值
```

**技巧 2：类型混淆**

```bash
# 发送不同类型的数据
POST /api/checkout
{"total": "0.01"}  # 字符串而非数字
{"total": null}
{"total": ""}
{"total": -100}  # 负数
```

**技巧 3：JSON 注入**

```bash
# 在 JSON 中注入额外字段
POST /api/checkout
{
    "items": [...],
    "total": 999.99,
    "total": 0.01  # 重复键
}
```

### 2.5.2 速率限制绕过

**技巧 4：分布式请求**

```bash
# 使用多个 IP/代理
# 每个 IP 低于阈值
```

**技巧 5：参数变换**

```bash
# 改变请求格式绕过检测
Content-Type: application/json
Content-Type: application/x-www-form-urlencoded
Content-Type: multipart/form-data
```

### 2.5.3 业务规则绕过

**技巧 6：边界值利用**

```bash
# 测试边界条件
quantity: 0       # 零值
quantity: -1      # 负值
quantity: 999999  # 超大值
quantity: 0.5     # 小数值
quantity: null    # 空值
```

**技巧 7：时序攻击**

```bash
# 利用时间窗口
# 在库存更新前完成支付
# 在价格变更前完成下单
```

# 第三部分：附录

## 3.1 业务逻辑测试检查清单

```
□ 价格是否在服务端重新计算
□ 数量是否有限购检查
□ 优惠券是否验证有效性和适用范围
□ 库存是否有并发控制
□ 订单状态是否可以跳过或回退
□ 是否有重复提交防护
□ 是否有速率限制
□ 是否有异常行为检测
□ 事务是否正确回滚
□ 是否有审计日志
```

## 3.3 修复建议

1. **服务端验证** - 所有关键计算在服务端进行
2. **业务规则编码** - 将业务规则硬编码到服务端逻辑中
3. **状态机验证** - 严格验证状态转换的合法性
4. **并发控制** - 使用锁或乐观锁处理并发
5. **事务管理** - 确保事务的原子性和一致性
6. **异常检测** - 监控和告警异常业务行为
7. **审计日志** - 记录所有关键业务操作
