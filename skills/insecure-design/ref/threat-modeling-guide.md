# 威胁建模指南

---

# 第一部分：文档概述

## 1.1 编写目的

本文档旨在为渗透测试人员和安全工程师提供一套标准化的威胁建模方法，帮助系统性地识别、评估和缓解系统设计中的安全威胁。

## 1.2 适用范围

本文档适用于系统设计阶段的安全评估、现有系统的安全审查、以及渗透测试前的信息收集阶段。

## 1.3 读者对象

本文件主要面向安全工程师、架构师、渗透测试人员、安全开发人员。

# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 威胁建模原理

威胁建模是一种结构化的方法，用于识别、评估和缓解系统中的潜在安全威胁。

**核心目标**：
- 识别潜在威胁
- 评估威胁风险
- 确定缓解措施
- 优先处理高风险

**本质问题**：
- 从攻击者视角思考
- 系统性分析攻击面
- 预防性安全设计

### 常见威胁建模方法

| 方法 | 描述 | 适用场景 |
|-----|------|---------|
| STRIDE | 微软提出的分类方法 | 通用系统设计 |
| PASTA | 以风险为中心 | 企业级应用 |
| VAST | 以可视化为中心 | 敏捷开发 |
| OCTAVE | 以资产为中心 | 组织风险评估 |
| Attack Trees | 树状分析 | 特定攻击场景 |

## 2.2 威胁建模流程

### 2.2.1 准备阶段

**步骤 1：确定范围**

```
定义建模范围：
- 系统边界
- 包含的组件
- 外部依赖
- 数据流范围
```

**步骤 2：组建团队**

```
团队成员：
- 架构师
- 开发人员
- 安全工程师
- 运维人员
- 业务代表
```

**步骤 3：收集资料**

```
收集资料：
- 系统架构图
- 数据流图
- 部署图
- 需求文档
- 合规要求
```

### 2.2.2 系统分解

**步骤 1：绘制数据流图（DFD）**

```
DFD 元素：
┌─────┐     外部实体（External Entity）
│ ○ │       过程（Process）
│ ═ │       数据存储（Data Store）
│ → │       数据流（Data Flow）
```

**示例：简单登录系统 DFD**

```
     ┌────────┐
     │ 用户   │
     └───┬────┘
         │ 登录请求
         ▼
     ┌───────────┐
     │ 登录处理  │
     └─────┬─────┘
           │ 查询
     ┌─────▼─────┐
     │ 用户数据库 │
     └───────────┘
```

**步骤 2：识别信任边界**

```
信任边界：
- 用户 ↔ 应用（不信任）
- 应用 ↔ 数据库（部分信任）
- 内部服务 ↔ 内部服务（过度信任）
```

**步骤 3：识别资产**

```
关键资产：
- 用户数据
- 认证凭证
- 业务数据
- 系统配置
- API 密钥
```

### 2.2.3 威胁识别（STRIDE）

**S - 欺骗（Spoofing）**

```
威胁问题：
- 攻击者能否冒充合法用户？
- 攻击者能否冒充其他系统？
- 认证机制是否足够？

缓解措施：
- 强认证机制
- 多因素认证
- 证书验证
```

**T - 篡改（Tampering）**

```
威胁问题：
- 数据能否被篡改？
- 代码能否被修改？
- 配置能否被更改？

缓解措施：
- 数字签名
- 完整性校验
- 访问控制
```

**R - 抵赖（Repudiation）**

```
威胁问题：
- 用户能否否认操作？
- 是否有审计日志？
- 日志是否可篡改？

缓解措施：
- 完整审计日志
- 日志保护
- 数字签名
```

**I - 信息泄露（Information Disclosure）**

```
威胁问题：
- 敏感数据是否加密？
- 是否有未授权访问？
- 错误信息是否泄露？

缓解措施：
- 数据加密
- 访问控制
- 错误处理
```

**D - 拒绝服务（Denial of Service）**

```
威胁问题：
- 系统能否被耗尽？
- 是否有速率限制？
- 是否有资源限制？

缓解措施：
- 速率限制
- 资源配额
- 弹性设计
```

**E - 权限提升（Elevation of Privilege）**

```
威胁问题：
- 能否越权访问？
- 能否提升权限？
- 权限检查是否充分？

缓解措施：
- 最小权限原则
- 权限验证
- 访问控制
```

### 2.2.4 威胁评估

**风险评估矩阵**

```
风险 = 可能性 × 影响

可能性等级：
1 - 极低
2 - 低
3 - 中
4 - 高
5 - 极高

影响等级：
1 - 极低
2 - 低
3 - 中
4 - 高
5 - 极高
```

**风险优先级**

```
风险值 = 可能性 × 影响

15-25: 高风险（立即处理）
8-14:  中风险（计划处理）
1-7:   低风险（接受或监控）
```

### 2.2.5 威胁缓解

**缓解策略**

```
1. 消除 - 完全移除威胁
2. 降低 - 减少可能性或影响
3. 转移 - 转移给第三方
4. 接受 - 接受风险存在
```

**缓解措施分类**

```
预防性控制：
- 认证授权
- 输入验证
- 加密

检测性控制：
- 日志审计
- 入侵检测
- 监控告警

纠正性控制：
- 备份恢复
- 应急响应
- 补丁管理
```

## 3.3 最佳实践

1. **早期介入** - 在设计阶段开始威胁建模
2. **持续更新** - 系统变更时更新模型
3. **团队协作** - 多角色参与建模
4. **聚焦重点** - 优先处理高风险威胁
5. **可操作** - 确保缓解措施可执行
6. **文档化** - 完整记录建模过程
