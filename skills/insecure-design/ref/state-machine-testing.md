# 状态机安全测试方法论

---

# 第一部分：文档概述

## 1.1 编写目的

本文档旨在为渗透测试人员提供一套标准化的状态机安全测试方法，帮助发现和利用状态转换设计中的安全缺陷。

## 1.2 适用范围

本文档适用于所有具有状态流转的业务系统，如订单系统、审批流程、支付系统、工单系统等。

## 1.3 读者对象

本文件主要面向执行渗透测试任务的安全工程师、安全分析师。

# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 状态机安全漏洞原理

状态机安全漏洞是指系统在状态定义、状态转换条件、状态验证等设计层面的缺陷，导致攻击者可以绕过正常状态流转达成恶意目的。

**核心概念**：
- 状态（State）
- 转换（Transition）
- 事件（Event）
- 守卫条件（Guard Condition）

**本质问题**：
- 状态转换验证不足
- 状态跳转条件缺陷
- 状态一致性检查缺失
- 并发状态处理缺陷

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-840 | 业务逻辑缺陷 |
| CWE-284 | 访问控制不当 |
| CWE-362 | 并发资源竞争 |

## 2.2 状态机分析

### 2.2.1 状态识别

**步骤 1：识别系统状态**

```
示例：订单系统状态
- 待支付（Pending Payment）
- 已支付（Paid）
- 已发货（Shipped）
- 已完成（Completed）
- 已取消（Cancelled）
- 退款中（Refunding）
- 已退款（Refunded）
```

**步骤 2：识别状态转换**

```
示例：订单状态转换

待支付 ──支付──→ 已支付
  │                  │
  │取消              │发货
  ↓                  ↓
已取消            已发货
                    │
                    │确认收货
                    ↓
                  已完成
```

**步骤 3：识别转换条件**

```
示例：转换条件
- 待支付 → 已支付：支付成功
- 已支付 → 已发货：商家发货
- 已发货 → 已完成：用户确认收货
- 任意状态 → 已取消：用户/商家取消（有条件）
```

## 2.3 测试方法

### 2.3.1 状态跳过测试

**测试目标**：验证是否可以跳过中间状态

```bash
# 场景：订单系统

# 测试 1：跳过支付状态
# 待支付 → 已发货（跳过已支付）

POST /api/order/update-status
{
    "order_id": "123",
    "status": "shipped"  # 从待支付直接到已发货
}

# 测试 2：跳过发货状态
# 已支付 → 已完成（跳过已发货）

POST /api/order/update-status
{
    "order_id": "123",
    "status": "completed"  # 从已支付直接到已完成
}

# 测试 3：跳过审批流程
# 提交 → 批准（跳过审核中）

POST /api/approval/update
{
    "approval_id": "456",
    "status": "approved"  # 从提交直接到批准
}
```

### 2.3.2 状态回退测试

**测试目标**：验证是否可以将状态回退到之前状态

```bash
# 场景：订单系统

# 测试 1：已完成 → 已支付

POST /api/order/update-status
{
    "order_id": "123",
    "status": "paid"  # 回退到已支付
}

# 测试 2：已发货 → 待支付

POST /api/order/update-status
{
    "order_id": "123",
    "status": "pending"  # 回退到待支付
}

# 测试 3：已批准 → 待审核

POST /api/approval/update
{
    "approval_id": "456",
    "status": "pending"  # 回退到待审核
}
```

### 2.3.3 非法状态转换测试

**测试目标**：验证是否阻止非法状态转换

```bash
# 场景：订单系统

# 测试 1：已取消 → 已支付

POST /api/order/update-status
{
    "order_id": "123",
    "status": "paid"  # 已取消订单不能支付
}

# 测试 2：已完成 → 退款中

POST /api/order/refund
{
    "order_id": "123"
    # 已完成订单可能不允许退款
}

# 测试 3：已退款 → 已支付

POST /api/order/update-status
{
    "order_id": "123",
    "status": "paid"  # 已退款订单不能变回已支付
}
```

### 2.3.4 并发状态转换测试

**测试目标**：验证并发状态转换的处理

```bash
# 场景：订单系统

# 测试 1：并发状态更新
# 同时发起多个状态更新请求

POST /api/order/update-status
{"order_id": "123", "status": "shipped"}

POST /api/order/update-status
{"order_id": "123", "status": "cancelled"}

POST /api/order/update-status
{"order_id": "123", "status": "refunded"}

# 检查最终状态是否一致

# 测试 2：并发支付和取消
# 用户支付同时商家取消

POST /api/order/pay
{"order_id": "123"}

POST /api/order/cancel
{"order_id": "123"}

# 检查最终状态
```

### 2.3.5 状态条件绕过测试

**测试目标**：验证状态转换条件是否可绕过

```bash
# 场景：订单系统

# 测试 1：未支付直接发货

POST /api/order/ship
{
    "order_id": "123"
    # 订单状态是待支付，但尝试发货
}

# 测试 2：未发货直接完成

POST /api/order/complete
{
    "order_id": "123"
    # 订单状态是已支付，但尝试完成
}

# 测试 3：条件参数篡改

POST /api/order/ship
{
    "order_id": "123",
    "force": true  # 尝试强制操作
}
```

### 2.3.6 状态一致性测试

**测试目标**：验证状态与相关业务数据一致性

```bash
# 场景：订单系统

# 测试 1：状态与支付金额一致性
# 已支付状态但支付金额为 0

GET /api/order/123
# 检查：status=paid, paid_amount=0

# 测试 2：状态与库存一致性
# 已取消订单但库存未恢复

GET /api/order/123
GET /api/product/456/stock
# 检查库存是否正确恢复

# 测试 3：状态与物流一致性
# 未发货状态但有物流信息

GET /api/order/123
GET /api/order/123/logistics
# 检查物流信息是否匹配
```

## 2.4 常见状态机漏洞模式

### 2.4.1 模式 1：状态枚举不完整

```
漏洞描述：
系统未定义所有可能状态，导致状态混乱

示例：
- 缺少"处理中"状态
- 缺少"异常"状态
- 缺少"超时"状态

测试方法：
- 识别所有可能的业务场景
- 检查是否有对应状态
- 测试异常场景的状态处理
```

### 2.4.2 模式 2：状态转换验证不足

```
漏洞描述：
状态转换时未验证前置条件

示例：
- 未验证支付成功就改为已支付
- 未验证发货就改为已发货
- 未验证权限就允许状态变更

测试方法：
- 测试跳过前置状态
- 测试条件不满足时转换
- 测试参数篡改绕过
```

### 2.4.3 模式 3：状态回退未限制

```
漏洞描述：
允许状态回退导致业务逻辑混乱

示例：
- 已完成订单回退到待支付
- 已批准申请回退到待审核
- 已发货订单回退到待发货

测试方法：
- 测试所有状态回退
- 检查回退后的业务影响
- 检查关联数据一致性
```

### 2.4.4 模式 4：并发状态更新冲突

```
漏洞描述：
并发状态更新导致状态不一致

示例：
- 同时支付和取消
- 同时发货和退款
- 同时完成和取消

测试方法：
- 并发发起状态更新
- 检查最终状态
- 检查数据一致性
```

### 2.4.5 模式 5：状态超时处理缺陷

```
漏洞描述：
状态超时处理不当导致问题

示例：
- 待支付订单未超时取消
- 超时后仍可支付
- 超时状态未通知

测试方法：
- 等待订单超时
- 测试超时后操作
- 检查超时处理逻辑
```

## 2.5 利用方法

### 2.5.1 0 元购攻击

```
攻击流程：
1. 创建订单（待支付状态）
2. 跳过支付直接发货
3. 确认收货完成订单
4. 商品到手未付款
```

### 2.5.2 退款欺诈

```
攻击流程：
1. 正常下单支付
2. 申请退款
3. 退款成功后状态回退
4. 再次申请退款
5. 重复获利
```

### 2.5.3 库存超卖

```
攻击流程：
1. 创建多个订单占用库存
2. 并发取消订单
3. 库存重复恢复
4. 库存数量异常
```

### 2.5.4 审批绕过

```
攻击流程：
1. 提交申请
2. 直接修改状态为批准
3. 跳过审批流程
4. 获取不当权限/资源
```

## 3.3 修复建议

1. **状态图设计** - 完整设计状态图
2. **转换验证** - 验证所有状态转换
3. **守卫条件** - 实现守卫条件检查
4. **状态持久化** - 使用数据库约束
5. **并发控制** - 使用锁或乐观锁
6. **状态审计** - 记录状态变更日志
7. **超时处理** - 实现超时自动处理
8. **异常处理** - 处理异常状态场景
