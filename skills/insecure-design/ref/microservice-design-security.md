# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 微服务安全漏洞原理

微服务安全漏洞是指微服务架构在服务通信、服务发现、配置管理、链路追踪等设计层面的缺陷。

**核心设计层面**：
- 服务间通信安全
- 服务发现与注册
- 配置中心安全
- 链路追踪与监控

**本质问题**：
- 服务间信任设计缺陷
- 身份传递设计不足
- 配置管理设计缺陷
- 边界防护设计不足

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-284 | 访问控制不当 |
| CWE-200 | 信息泄露 |
| CWE-311 | 加密存储缺失 |

## 2.2 攻击常见于哪些业务场景

| 业务场景 | 功能示例 | 风险点描述 |
|---------|---------|-----------|
| 服务通信 | HTTP/gRPC 调用 | 未加密、未认证 |
| 服务发现 | 服务注册/发现 | 恶意服务注册 |
| 配置管理 | 配置中心 | 配置泄露/篡改 |
| API 网关 | 请求路由 | 认证绕过、越权 |
| 链路追踪 | 请求追踪 | 敏感信息泄露 |
| 服务网格 | Sidecar 代理 | 配置篡改 |
| 容器编排 | Kubernetes | 配置泄露、权限提升 |
| 消息队列 | 事件驱动 | 消息伪造、重放 |

## 2.3 漏洞发现方法

### 2.3.1 架构分析

**步骤 1：绘制服务架构图**

```
典型微服务架构：
┌─────────────┐
│  API Gateway │
└──────┬──────┘
       │
┌──────┼──────┬────────┐
│      │      │        │
┌▼─────▼─┐ ┌─▼──────┐ ┌▼──────┐
│用户服务│ │订单服务│ │支付服务│
└────────┘ └────────┘ └───────┘
```

**步骤 2：识别通信方式**

```
服务间通信：
- 同步：HTTP/REST、gRPC
- 异步：消息队列（Kafka、RabbitMQ）
- 服务网格：Istio、Linkerd
```

**步骤 3：识别信任边界**

```
信任边界：
- 外部 → API 网关（不信任）
- API 网关 → 内部服务（部分信任）
- 内部服务 → 内部服务（过度信任）
```

### 2.3.2 服务枚举

```
服务发现方式：
- 服务注册中心（Consul、Eureka）
- Kubernetes Service
- DNS 服务发现
- 配置文件
```

### 2.4.1 服务间通信测试

```bash

# 绕过 API 网关

# 2. 服务端口扫描
# 发现未暴露的服务端口

# 3. 通信加密测试
# 检查是否使用 HTTPS/mTLS
```

### 2.4.2 服务认证测试

```bash
# 场景：服务间认证

# 1. 伪造服务身份
POST http://user-service/api/validate
Authorization: Bearer $FORGED_TOKEN
# 使用伪造的服务令牌

# 2. 服务令牌重放
# 重放捕获的服务间请求

# 3. 无认证访问
GET http://order-service/api/orders
# 检查是否需要认证
```

### 2.4.3 配置中心测试

```bash
# 场景：配置管理

# 1. 未授权访问配置
```

## 2.5 漏洞利用方法

### 2.5.1 横向移动攻击

```bash
# 利用服务间信任

# 1. 入侵边缘服务
# 2. 利用服务间信任访问其他服务
# 3. 逐步渗透到核心服务
```

```bash
# 利用配置管理漏洞

```

# 第三部分：附录

## 3.1 微服务安全测试检查清单

```
□ 服务间通信是否加密
□ 服务间是否有认证
□ API 网关是否可绕过
□ 配置中心是否保护
□ 服务发现是否安全
□ 链路追踪是否泄露
□ 容器配置是否安全
□ 消息队列是否安全
□ 服务网格配置是否正确
□ 是否有网络分段
```

## 3.3 修复建议

1. **mTLS** - 服务间通信使用 mTLS
2. **服务认证** - 服务间请求需要认证
3. **网络策略** - 实施网络分段
4. **配置保护** - 配置中心需要认证和加密
5. **API 网关** - 所有外部请求通过网关
6. **最小权限** - 服务使用最小权限
7. **审计日志** - 记录服务间通信
8. **安全监控** - 监控异常服务行为
