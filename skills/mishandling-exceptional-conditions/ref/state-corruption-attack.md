# 第二部分：核心渗透技术专题

## 2.1 技术介绍

### 漏洞原理

状态腐败攻击是指利用系统在异常条件下（如网络中断、超时、崩溃）未能正确维护状态一致性，导致数据损坏或资金丢失。

**本质问题**：
- 事务未正确回滚
- 状态更新非原子性
- 缺少补偿机制

### 常见 CWE 映射

| CWE 编号 | 描述 |
|---------|------|
| CWE-636 | 未安全失败 (Failing Open) |
| CWE-362 | 并发执行共享资源的同步问题 |
| CWE-820 | 缺少同步 |

## 2.2 攻击常见于哪些业务场景

| 场景 | 风险点 | 潜在危害 |
|-----|-------|---------|
| 支付系统 | 部分事务提交 | 资金丢失/重复 |
| 订单系统 | 订单状态不一致 | 超卖/欠货 |
| 库存系统 | 库存扣减失败 | 库存不准确 |
| 账户系统 | 余额更新失败 | 余额错误 |
| 积分系统 | 积分增减不一致 | 积分错误 |

## 2.4 漏洞利用方法

### 2.4.1 资金丢失攻击

```
场景：银行转账

正常流程：
1. 检查 A 账户余额充足
2. 扣减 A 账户 100 元
3. 增加 B 账户 100 元
4. 记录交易日志

攻击流程：
1. 发起转账请求
2. 在步骤 2 和 3 之间中断连接
3. 步骤 2 已提交，步骤 3 未执行
4. 结果：A 账户扣款，B 账户未入账
5. 100 元"消失"
```

### 2.4.2 双重支付攻击

```
场景：电子钱包支付

攻击流程：
1. 发起支付请求（余额 100，支付 100）
2. 在扣款前复制请求
3. 快速发送两个请求
4. 如果检查与扣款非原子
5. 两个请求都通过余额检查
6. 结果：支付 200，余额 -100
```

### 2.4.3 订单状态腐败

```
场景：电商订单

攻击流程：
1. 下单并支付
2. 在订单状态更新前取消
3. 如果状态更新非原子
4. 可能：已支付 + 已取消
5. 或：未支付 + 已发货
```

### 2.4.4 库存超卖

```
场景：限量商品

攻击流程：
1. 库存 10 件
2. 并发 100 个下单请求
3. 每个请求检查库存 > 0
4. 所有请求都通过检查
5. 库存扣减非原子执行
6. 结果：卖出 100 件，库存 -90
```

## 3.3 修复建议

1. **事务完整性** - 确保事务原子性
2. **补偿机制** - 实现 Saga 等补偿模式
3. **幂等性** - 确保操作可重复执行
4. **状态机验证** - 严格验证状态转换
5. **分布式锁** - 使用锁保证并发安全
